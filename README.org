* ob-futur.el
Adds =:futur-async= header argument to elisp source blocks. Two modes, dispatched
by value. Requires ~ob-futur-mode~ to be enabled.

- =:futur-async yes= ::

Run the body in a batch Emacs subprocess via ~futur-elisp--funcall~. Fully
non-blocking. No access to current Emacs state (buffers, windows, dynamic
bindings). Return values must be print/read round-trippable. Use for CPU-bound
work.

The subprocess has no access to the current buffer. To pass buffer data, use
Org's =:var= header which evaluates in the current Emacs before the body ships to
the subprocess:

#+begin_src elisp
;;#+begin_src elisp :futur-async yes :var text=(buffer-string) dir=default-directory
(format "Buffer has %d chars in %s" (length text) dir)
;;+end_src
#+end_src

- =:futur-async bind= ::

Evaluate the body in the current Emacs. Expect a futur object as return value,
resolve it via ~futur-bind~. Full access to Emacs state (buffers,
default-directory~, dynamic bindings at eval time). Use for process-based /
network-based async operations built on futur constructors like
futur-shell-command-to-string~, ~futur-process-call~, or ~futur-url-retrieve~.

If the body returns a plain (non-futur) value, falls back to synchronous
formatting via the stock ~org-babel-result-cond~ path.

Both modes insert a placeholder =Executing asynchronously...= immediately, then
replace it with the resolved result when the async operation completes.
Incompatible with =:results output=.

CPU-bound example (subprocess, non-blocking):

#+begin_src elisp :export none
;;#+begin_src elisp :futur-async yes
(let ((primes nil) (n 2))
  (while (< (length primes) 5000)
    (when (cl-loop for d from 2 to (cl-isqrt n)
                   never (zerop (% n d)))
      (push n primes))
    (cl-incf n))
  (format "Found %d primes, largest: %d"
          (length primes) (car primes)))
;;#+end_src
#+end_src

Process-based example (current Emacs, futur resolved via bind):

#+begin_src elisp :export none
;;#+begin_src elisp :futur-async bind
(futur-shell-command-to-string "sleep 2 && echo done")
;;,#+end_src
#+end_src

Passing buffer context to subprocess via =:var=:

#+begin_src elisp :export none
;;#+begin_src elisp :futur-async yes :var lines=(count-lines (point-min) (point-max))
(format "Buffer had %d lines at eval time" lines)
;;#+end_src
#+end_src

* futur-shell.el
attempt at getting same functionality as in =async-shell-command=
with futur, not looking for full parity with the [[info:emacs#Shell][Shell]] utilities, just looking
to be able to execute asynchronously and display the progress on a buffer.

I'm also trying to use it to find more usecases. For example as I was trying
to integrate a cli tool through /futur-shell/, I realized I didn't have a way of
executing futur commands in org-mode elisp sourceblocks and get the callback
output in the results, so ob-futur is an attempt at that.

* IDEA - Adapt dired-async to use futur
dired-async already exists, but it might be useful to adapt it to futur just to
learn more usage patterns. It does solve async issues in a similar manner as
futur, most of it's async-* functions can be mapped to futur ones.

For example:

| emacs-async              | futur equivalent                              |
|--------------------------+-----------------------------------------------|
| ~async-start~ with lambda  | ~futur-elisp--funcall~ with lambda              |
| ~async-inject-variables~   | Lexical capture in closure sent to subprocess |
| ~async-when-done~ sentinel | ~futur-bind~ callback                           |
| etc                      |                                               |

Main advantage of using futur here would be composability, the /futur/ returned by
the file operation can be chained through /futur-let*/ so things like /copy then
byte-compile/ could be done without the nested callbacks... maybe?

No idea how to deal with TRAMP, maybe that would be the main benefit of trying

the modeline that reports progress on an operation, say If I delete a folder
